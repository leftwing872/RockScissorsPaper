🛠️ Agones System (클러스터 두뇌, agones-system 네임스페이스)
Controller가 GameServer/Fleet 리소스를 감시하며 파드를 만들고, 노드 포트를 배정하고, 죽은 서버를 재시작한다. 각 파드에 사이드카를 자동으로 끼워 넣는 것도 이 친구.
내 게임 서버 컨테이너 Agones가 자동 주입하는 사이드카 Ready = 매치 대기중 Allocated = 매치 잡힘

3. 포트는 어떻게 연결되나 (네가 헷갈렸던 부분)

클라이언트 게임 플레이어
→접속
노드IP : 7361 hostPort (동적 배정)
→매핑
컨테이너 : 7777 containerPort (코드 고정)
핵심: Dockerfile / YAML의 containerPort: 7777은 컨테이너 안쪽 고정 포트(= 코드의 port = 7777). 외부에서 실제 접속하는 포트는 Agones가 노드에서 겹치지 않게 자동으로 골라준 hostPort(예: 7361)다. 그래서 같은 노드에 7777짜리 서버를 여러 개 띄워도 충돌하지 않는다.
1
YAML에 portPolicy: Dynamic + containerPort: 7777 지정 → Agones가 노드 포트(예 7361)를 자동 배정
2
매치메이커가 Allocation 요청 → 응답으로 address: 노드IP + port: 7361 을 받음
3
클라이언트는 노드IP : 7361 로 TCP 접속 → 노드가 컨테이너 :7777 로 전달 → 내 서버가 받음
웹 서비스와 달리 게임 서버는 Service/Ingress/로드밸런서를 거치지 않는다. 저지연을 위해 클라이언트가 노드에 직접 붙고, Kubernetes의 hostPort(iptables 매핑)를 쓴다. 별도 프록시 계층이 없어 지연이 거의 없다.

4. 한 서버의 일생 (코드 기준)

1
부팅 → ConnectAsync() 로 사이드카에 연결
2
준비 완료 → ReadyAsync() "나 매치 받을 수 있어" → 상태 Ready
3
살아있음 증명 → 2초마다 HealthAsync() 핑 (백그라운드 루프)
4
할당됨 → 매치메이커가 잡으면 상태 Allocated · 클라이언트 2명 접속
5
한 판 종료 (GameOver) → ShutdownAsync() → 파드 정리 → Fleet이 새 Ready 보충
